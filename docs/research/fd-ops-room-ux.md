## 1. Cybersyn Operations Room (Chile, 1971-1973)

### Sources

- Eden Medina, *Cybernetic Revolutionaries: Technology and Politics in Allende's Chile* (MIT Press, 2011). [PDF](https://uberty.org/wp-content/uploads/2015/10/Eden_Medina_Cybernetic_Revolutionaries.pdf)
- Sebastian Vehlken, "Operative communication: project Cybersyn and the intersection of information design, interface design, and interaction design," *AI & Society* 37 (2022): 1131-1152. [Springer](https://link.springer.com/article/10.1007/s00146-021-01346-2)
- [Cabinet Magazine, Issue 46 - Eden Medina on Cybersyn](https://www.cabinetmagazine.org/issues/46/medina.php)
- [99% Invisible Episode: Project Cybersyn](https://99percentinvisible.org/episode/project-cybersyn/)
- [Cybersyn on Metaphorum (Stafford Beer archive)](https://metaphorum.org/staffords-work/cybersyn)
- MIT Exhibition reconstruction (2023): [MIT News](https://news.mit.edu/2023/designing-revolution-1002)

### Concrete Design Principles

**Hexagonal geometry with no table.** The room was hexagonal (originally intended circular), providing five wall surfaces for display screens plus an entrance. The designers explicitly removed any table to prevent paper-shuffling and force attention toward the screens. Seven white fiberglass swivel chairs with orange cushions sat in the center, arranged for face-to-face discussion while all walls remained visible. The choice of swivel chairs was deliberate -- Bonsiepe's team considered them optimal for creative group decision-making.

**Armrest-embedded controls, no keyboard.** Each chair's fiberglass armrest contained geometric push-buttons that allowed any occupant to control the wall screens. A "hold" button gave temporary exclusive control. The design deliberately avoided keyboards and technical input devices, ensuring that factory managers without computer experience could navigate the system. This was one of the earliest instances of what we now call "direct manipulation" -- users created their own nonlinear path through data, predating hypertext by two decades.

**Datafeed screens with back-projection.** Four screens on one wall were housed in organic-curved fiberglass cabinets. One large screen above three smaller screens showed the button combinations needed to change the data below. Behind the screens, slide projectors back-projected hand-drawn flow diagrams and statistical graphs, simulating flat-panel displays with 1970s technology.

**Algedonic (exceedance) signals.** Two screens on another wall recorded "algedonic signals" -- early warnings generated by the Cyberfilter software, which performed time-series statistical analysis to forecast incipient instability in factory production data. If one management level did not remedy a problem within a defined time interval, the alert escalated to the next level. This is a direct ancestor of modern alert escalation systems.

**Operative communication, not persuasive.** Gui Bonsiepe, trained at the Ulm School of Design, distinguished sharply between "persuasive communication" (advertising, consumer influence) and "operative communication" (enabling effective action). The Opsroom was purely operative: language-independent pictograms, visual grammars, and information-dense displays designed for decision velocity, not aesthetic appeal. Together with Tomas Maldonado, Bonsiepe had earlier created icon-based visual grammars for Olivetti computers (1960-61) that directly influenced the Opsroom's visual language.

**Viable System Model hierarchy.** The Opsroom was the apex of Stafford Beer's Viable System Model applied to Chile's nationalized economy. Data flowed upward through five recursive levels of organization, with the Opsroom as the "brain" that only intervened when lower levels failed to self-correct. The entire design assumed that most problems would be resolved before reaching the room.

### What Survived as Best Practice

- **Exception-driven attention**: Display the normal state quietly; surface only deviations. The algedonic escalation model is now standard in incident management (PagerDuty, OpsGenie).
- **No-keyboard interaction for executive-level dashboards**: Modern executive dashboards use click/tap navigation rather than query languages.
- **Shared immersive environment over individual screens**: The hexagonal room anticipated modern "mission control" video walls where everyone sees the same macro state.
- **Nonlinear data exploration**: The armrest-button navigation model foreshadowed dashboard drill-down and hyperlinked data exploration.

### Mapping to Meadowsyn

| Cybersyn Pattern | Meadowsyn Application |
|---|---|
| Hexagonal shared display space | Full-width "hero" overview panel visible without scrolling; all agent states at a glance |
| Algedonic escalation signals | Tiered alert system: agent anomaly -> team-level warning -> factory-level alarm, with auto-escalation timers |
| Armrest buttons / no keyboard | Primary interaction via click/hover on agent tiles, not CLI or search; zero-learning-curve navigation |
| Operative communication | Suppress decorative elements; every pixel must encode operational state |
| Absence of paper/table | No static text panels or documentation embeds in the live view; reference material in a separate pane |
| Viable System Model hierarchy | Recursive zoom: factory overview -> team clusters -> individual agent -> bead detail |

## 2. NASA Mission Control Console Layout

### Concrete UI Patterns

**Tiered spatial hierarchy (front room / back room).** The Flight Control Room (FCR, "front room") contains 15-20 console positions arranged in ascending rows. The Flight Director (FLIGHT) sits near the center for optimal coordination. The fourth row, "The Trench," contains Booster, Retrofire Officer, Flight Dynamics Officer, and Guidance Officer -- the trajectory specialists who need the tightest real-time awareness. Behind the FCR, Staff Support Rooms ("back rooms") provide deeper system expertise. This two-tier structure separates integration decisions from specialist analysis. Back room controllers support the front room and handle commanding, procedures, and deep diagnosis.

**Big Board shared displays.** Five large screens at the front of the room (the center screen 20 feet wide, 10 feet tall; side screens 10x7.5 feet) show information of interest to everyone: live camera feeds, ground-track orbital maps, alarm/warning status, countdown clocks, and communication link status. During Apollo, these used Eidophor projectors, servo-controlled styluses cutting through metal-coated slides for real-time data plots, and layered glass/metal stencils with dichroic filters for icon overlays. The IBM System/360 drove the underlying data processing. This hybrid electromechanical system delivered crispness "far beyond what 1960s video could achieve" and remained in use for ~20 years until digital replacements arrived in 1989.

**Go/No-Go poll protocol.** Before major events (launch commit, orbit entry/exit, abort decisions), the Flight Director polls each console position sequentially. Each controller reports "Go" or "No-Go" based on their subsystem status. A single "No-Go" can halt the entire operation. The Launch Readiness Review occurs no later than 1 day before launch; the Final Commit-to-Launch Poll occurs at T-5 minutes. This protocol forces explicit status declaration from every responsible party, preventing assumption-based failures.

**Flight Rules as pre-computed decision trees.** NASA's Flight Rules document is the "operational bible" -- pre-planned responses for every anticipated anomaly, organized by subsystem (GNC, Propulsion, Electrical Power, Mechanical, Communications, Data Systems, Robotics, Life Support). The rules minimize real-time discussion by providing deterministic responses for known failure modes, reserving human judgment for novel situations only.

**Alarm triage hierarchy.** The Caution and Warning system monitors critical spacecraft parameters. Out-of-tolerance conditions illuminate a status light identifying the abnormality, activate master alarm lights, and send audio tones to headsets. This is a three-layer triage:
1. **Advisory** -- informational, no immediate action required
2. **Caution** -- degraded condition, investigate and prepare contingency
3. **Warning** -- immediate threat to crew/mission safety, execute flight rule response

**Anomaly detection workflow.** When an anomaly is detected (by spacecraft or ground systems), the appropriate operations team members are alerted immediately via text messaging. If the anomaly matches a known pattern, the automated system initiates the predetermined response. If novel, the front room controller coordinates with back room specialists, who have deeper telemetry access and wider historical context. The operations and engineering communities are spatially separated with distinct scopes of authority.

### Mapping to Meadowsyn

| NASA Pattern | Meadowsyn Application |
|---|---|
| Tiered rows (front/back) | Two-layer dashboard: summary bar (factory health, throughput, blockers) + detail panels (per-agent telemetry) |
| Big Board shared display | Persistent top banner showing aggregate factory metrics: active agents, completion rate, error rate, cost burn |
| Go/No-Go poll | "Readiness check" before deploying a batch of agents: each subsystem (API keys, repo access, context loaded) reports green/red |
| Flight Rules | Pre-defined runbooks for common agent failures (context overflow, API timeout, test flake) shown inline when triggered |
| Three-tier alarm hierarchy | Advisory (agent idle >5 min) / Caution (agent retrying, cost spike) / Warning (agent stuck, infinite loop, safety violation) |
| Front room / back room separation | Overview dashboard for factory managers; deep-dive panel for agent developers with full logs and trace data |

### Key Standards and Their Recommendations

#### NUREG-0700 (NRC, first published 1981, revised through Rev. 3)

Created after the Three Mile Island accident (1979). Organized in four parts:

- **Part I**: Basic HSI elements -- information displays, user-interface interaction and management, analog displays and controls. These are building blocks.
- **Part II**: HSI systems -- alarm system, safety parameter display system (SPDS), group-view display system, soft control system, computer-based procedure system, automation system, communication system.
- **Part III**: Integration and management guidelines.
- **Part IV**: Evaluation and testing.

Key alarm system principles:
- Alarms must support *detection*, *comprehension*, and *response* as distinct cognitive phases
- The SPDS provides an integrated overview of critical safety functions, continuously displayed on the large display panel
- Degraded sensor conditions must be distinguishable from process disturbances to prevent misdiagnosis
- Soft controls (touchscreen, mouse-based) introduce new error modes: navigation errors, mode confusion, context-dependent behavior that differs from spatially-dedicated physical controls

#### IEC 62241 (2004): Nuclear Plant Alarm Functions and Presentation

Establishes international requirements for alarm systems in nuclear main control rooms. Addresses five specific problems:
1. Omission of important alarms
2. Delay in detecting important alarms
3. Increased workload affecting other operational activities
4. Inattention to frequently activated "nuisance alarms"
5. Confusion from misunderstanding alarm relationships and importance

Recommends alarm functions be integrated into the main control room HMI system rather than implemented as a standalone system.

#### EEMUA 191 (Ed. 4, 2024): Alarm Systems Guide

The globally accepted benchmark for alarm management. Quantitative targets:

| Metric | Target | Overloaded |
|---|---|---|
| Average alarm rate (normal ops) | < 1 per 10 minutes | > 2 per 10 minutes |
| Peak alarm rate (10 min after upset) | < 10 per 10 min per operator | > 20 per 10 min |
| Standing (stale) alarms | < 10 | > 25 |
| Disabled/inhibited alarms | < 30 | > 50 |
| Alarm priority distribution | 10% high / 20% medium / 70% low | -- |

Key concepts:
- **Alarm flood**: >10 alarms annunciating in a 10-minute period (also ISA-18.2 definition)
- **Standing alarms**: Alarms that remain active for extended periods become "wallpaper" -- operators stop seeing them
- **Alarm rationalization**: Systematic review of every alarm against a philosophy document to confirm it is necessary, correctly prioritized, and has a defined operator response
- **Alarm shelving**: Operator manually suppresses an alarm temporarily, with automatic restoration after a preset time

#### ISA-18.2 (ANSI/ISA-18.2-2016): Alarm Management Lifecycle

Defines a lifecycle model: Philosophy -> Identification -> Rationalization -> Detailed Design -> Implementation -> Operation -> Maintenance. Alarm states follow a formal state model with transitions between:
- Normal / Active
- Unacknowledged / Acknowledged
- Suppressed by Design / Shelved
- Out of Service

**Suppressed by Design** uses process context (plant state, operating mode) to automatically suppress alarms that are not relevant in the current state -- e.g., suppressing low-level alarms during a controlled shutdown where low levels are expected.

#### ISA-101 HMI Display Hierarchy

Four-level navigation structure:

| Level | Name | Purpose | Access Frequency |
|---|---|---|---|
| L1 | Overview | Entire process at a glance; quick scan for major issues | Continuous |
| L2 | Unit Operation | Specific area or functional unit detail | Frequent |
| L3 | Task Detail | Major control modules, individual loops | As needed |
| L4 | Diagnostics and Settings | Configuration, maintenance, troubleshooting | Rare |

Critical principle: **shallow navigation**. The ASM Consortium demonstrated that all L2 displays must be accessible from anywhere in the HMI (not buried behind deep navigation trees). During abnormal situations, deep navigation wastes critical time.

#### ASM Consortium High-Performance HMI Findings

Founded 1994 by Honeywell, ABB, Rockwell. Key research results:
- 62% of abnormal situations are precipitated by operator actions
- Alarm floods during upset conditions average 300+ alarms/hour
- Traditional HMI designs require 3-5 seconds of cognitive interpretation per screen
- ASM-compliant HMI demonstrated: 41% faster task completion, 36% more accurate diagnoses, **380% greater detection of process disturbances before first alarm**

Color design rules:
- Color used sparingly, reserved for abnormal conditions
- Bright intense colors for alarms ONLY; never reuse alarm colors for normal state indicators
- Gray-scale for normal operations; the display should be "boring" during normal state
- Differentiate: live data vs. static text vs. input fields using color weight, not hue

### Mapping to Meadowsyn

| Nuclear HMI Pattern | Meadowsyn Application |
|---|---|
| EEMUA alarm rate targets | Rate-limit notifications: max 1 alert/10 min normal, throttle during incident to prevent flood |
| Alarm priority distribution (10/20/70) | Classify agent events: 10% critical (safety, cost), 20% important (blockers, failures), 70% informational (progress, state changes) |
| Standing alarm elimination | Auto-dismiss stale alerts after defined period; badge count shows how many are aging |
| Alarm shelving with auto-restore | "Snooze" button on agent alerts with mandatory 30-min auto-restore |
| ISA-101 L1-L4 hierarchy | Factory overview (L1) -> Team/project view (L2) -> Agent detail (L3) -> Logs/traces (L4) |
| Shallow navigation (ASM) | Every agent detail view accessible in 1 click from any screen |
| Color only for alarms | Agent tiles are gray-scale during normal ops; color appears only for state changes and alerts |
| Detection before alarm (380% improvement) | Trend indicators on agent tiles showing trajectory (slowing down, cost accelerating) before thresholds trip |
| SPDS safety overview | Persistent "factory health" bar with critical function indicators always visible |
| State-based suppression (ISA-18.2) | Suppress "agent idle" alerts when factory is in "paused" or "maintenance" mode |

### Concrete UI Patterns

**Data block content and positioning.** Each tracked aircraft displays a "data block" (label) connected to the position symbol by a "leader line." Minimum content: callsign/beacon code and Mode C altitude. Leader lines must not be displayed within +/- 90 degrees of the speed vector (to avoid confusion with trajectory). Controllers can manually reposition data blocks as a memory aid.

**Automatic label overlap avoidance.** ATC systems employ real-time algorithms that continuously evaluate data block positions and reposition them to avoid overlapping other labels, position symbols, and speed vectors. The algorithm finds feasible positions by varying angle and radius from the target position, creating "clutter-free" placement. This is computationally analogous to force-directed graph layout but with hard constraints on leader-line angles.

**Conflict Alert (CA) and Short-Term Conflict Alert (STCA).** When predicted separation between two aircraft falls below minima within a ~2-minute look-ahead window, the system generates an alert:
- "CA" (Collision Alert) or "LA" (Low Altitude Alert) text appears on top of the data block, blinking
- STCA alert colors the aircraft label red, appearing 50-90 seconds before loss of separation
- Controllers must "evaluate the reason for the alert without delay and take appropriate action"
- Conflict Suppress (CO) function allows suppression between specific aircraft pairs (e.g., during formation flying or air refueling)

**TCAS display color hierarchy.** The Traffic Collision Avoidance System uses a strict four-level color scheme:
1. **White/cyan diamond** (filled) -- proximate traffic, no threat
2. **Amber/yellow circle** (filled) -- Traffic Advisory (TA), within 900 ft vertically
3. **Red square** (filled) -- Resolution Advisory (RA), immediate collision threat
4. **Green band** on VSI -- "fly to" vertical speed region; **Red band** -- vertical speed to avoid

Traffic Advisories appear 35-48 seconds before predicted collision area entry.

**Altitude filtering.** Sectors configure altitude limits to display Mode C targets within their assigned stratum plus buffers (1,200 ft above/below for 1,000-ft separation areas; 2,200 ft for 2,000-ft areas). This filters out irrelevant traffic while maintaining safety margins.

**Sector handoff protocol.** As an aircraft approaches a sector boundary, the data block flashes on the current controller's display. Simultaneously, the receiving sector's display picks up the signal and flashes to alert the receiving controller. The handoff is a deliberate bilateral acknowledgment: initiating controller starts the handoff, receiving controller accepts, and radar identification transfers.

**Declutter techniques.** Research has identified four primary declutter methods:
1. **Dimming** -- reduce brightness of less-threatening symbols (25% faster response to threats in studies)
2. **Dotting** -- replace solid symbols with dotted outlines
3. **Small-sizing** -- reduce symbol/label size for lower-priority traffic
4. **Removal** -- completely hide non-essential targets

Of these, dimming yielded the best search performance in computational modeling studies. An automated declutter algorithm that continuously evaluates threat levels and dims non-threatening aircraft improved response timeliness by 25%.

**Scale and customization by role.** Planner controllers use wider views (larger area, smaller symbols); executive controllers use zoomed-in views focused on their sector. Label filtering has safety restrictions: mandatory display for planned entries, proximity aircraft, and emergency squawk codes (7500/7600/7700). Standard aspect ratio is 1:1 (square), though some facilities have adopted 16:9 widescreen.

### Mapping to Meadowsyn

| ATC Pattern | Meadowsyn Application |
|---|---|
| Data block with leader line | Agent cards with connector lines to their current work item (repo, issue, PR) |
| Automatic label overlap avoidance | Force-directed layout algorithm for agent tiles, preventing overlap as agents multiply |
| Conflict Alert (CA) flashing | Flashing indicator when two agents are working on the same file/issue (collision risk) |
| TCAS 4-level color hierarchy | Agent threat levels: white (nominal), amber (degraded), red (critical/stuck), green (completing successfully) |
| Altitude filtering | Filter agents by project, team, status; hide irrelevant agents from current operator's scope |
| Sector handoff flashing | Visual handoff animation when an agent passes work to another agent (PR review, context transfer) |
| Dimming for declutter | Idle/waiting agents rendered at 40% opacity; active agents at full brightness |
| Scale by role | Supervisor sees all agents in overview; developer sees only their team's agents zoomed in |
| 2-minute look-ahead | Predictive indicators: "agent likely to hit context limit in ~5 min" based on token consumption rate |
| Mandatory display for emergencies | Safety-critical events (infinite loop, credential exposure) always visible regardless of filter state |

## 5. Dark-Room Readability

### ISO 11064 Control Room Environment Standards

- **Minimum illuminance**: 200 lux recommended (compare: softly-lit living room ~50 lux; news studio ~1000 lux)
- **Unified Glare Index (UGI)**: must be below 19
- **Lighting types**: ambient (indirect, general), task (localized for documents), therapeutic (supporting circadian rhythm in 24/7 operations)
- **Key balance**: decrease contrast between work surfaces and surrounding areas; avoid direct light on screens; use indirect ambient lighting
- **Covers**: thermal environment, air quality, lighting, acoustics, vibration, aesthetics

### WCAG Contrast Requirements for Dark Mode

- **Normal text**: minimum 4.5:1 contrast ratio against background
- **Large text** (18pt+ or 14pt bold): minimum 3:1
- **UI components** (buttons, input fields, progress bars): minimum 3:1 against adjacent colors
- **Non-text elements** (meaningful graphical objects): minimum 3:1
- WCAG does not grant exceptions for dark mode -- both themes must independently meet all contrast requirements
- `prefers-color-scheme` media query should be respected; never override user's OS-level preference

### Chromaticity Choices for Low-Ambient Conditions

**Avoid pure black backgrounds.** Pure black (#000000) causes:
- "Halation effect" -- text appears to bleed/blur at edges on OLED screens due to extreme contrast
- Eye strain from pupil constantly readjusting between bright text and zero-luminance background
- Loss of perceived depth hierarchy

**Material Design 3 dark theme reference values:**
- Base surface: `#121212` (not pure black)
- Elevation overlays: primary color at increasing opacity (5% -> 16%) as elevation increases
- Overlay formula example: `#121212` + `#FFFFFF` at 9% opacity = `#272727`
- Tonal elevation system: higher surfaces use a more prominent tone rather than traditional shadows

**High-Performance HMI "Going Gray" palette:**
- Background: light gray (for lit control rooms) or dark gray `#2D2D2D`-`#3C3C3C` (for dark rooms)
- Static elements: dark gray `#A0A0A4` (Gray 160) outlines
- Live data: slightly brighter than static text to create perceived "glow"
- Equipment states: dark gray = scheduled; medium gray = running normal
- Alarm colors only: red (Priority 1), amber (Priority 2), yellow (Priority 3) -- never used for normal states
- Goal: normal operation display should be "somewhat boring"

**Desaturation principle.** In dark themes, saturated colors cause visual vibration against dark backgrounds. Desaturate all non-alarm colors by 20-40%. Only alarm indicators use full saturation, creating an automatic attention gradient.

**CSS implementation guidance:**
```css
/* Control-room-derived dark theme foundation */
:root {
  --surface-0: #121212;    /* Lowest elevation */
  --surface-1: #1E1E1E;    /* Cards, panels */
  --surface-2: #232323;    /* Elevated elements */
  --surface-3: #272727;    /* Modal backgrounds */
  --surface-4: #2C2C2C;    /* Floating elements */
  --text-primary: #E0E0E0;  /* Off-white, not pure white */
  --text-secondary: #A0A0A0; /* Muted labels */
  --text-disabled: #666666;  /* Inactive elements */
  --alarm-critical: #FF1744; /* Full saturation, Priority 1 */
  --alarm-warning: #FF9100;  /* Full saturation, Priority 2 */
  --alarm-caution: #FFD600;  /* Full saturation, Priority 3 */
  --nominal: #4CAF50;        /* Desaturated green, used sparingly */
  --border-subtle: #333333;  /* Barely visible structure */
}

@media (prefers-color-scheme: dark) {
  /* Respect OS preference automatically */
}
```

### Mapping to Meadowsyn

| Dark-Room Pattern | Meadowsyn Application |
|---|---|
| Off-black base (#121212) | Default background for all Meadowsyn views; never pure black |
| Tonal elevation hierarchy | Agent cards at surface-1, modals at surface-3, creating depth without shadows |
| Color reserved for alarms only | Agent tiles are gray-scale during normal ops; color appears only for state changes and alerts |
| Desaturated palette | All status colors desaturated 30% except alarm indicators |
| 4.5:1 contrast minimum | All text meets WCAG AA; automated contrast checking in design system |
| Halation avoidance | Off-white text (#E0E0E0) on off-black; never #FFFFFF on #000000 |
| prefers-color-scheme | Respect user's OS setting; Meadowsyn is dark-first but supports light mode |
| ISO 11064 anti-glare | No pure-white backgrounds; no large bright areas that cause reflections on nearby screens |
| 200 lux reference | Design optimized for dimmed-monitor NOC environments, not brightly-lit offices |

### What Was Lost

#### 1. The "Big Board" Overview (Keyhole/Soda-Straw Effect)

**Before:** Walking into a control room 20 years ago, you would see an entire wall with single-loop controllers, strip chart recorders, circular charts grouped by function. Operators could "see" their entire scope of responsibility simultaneously. The spatial arrangement itself encoded process relationships -- physically adjacent instruments meant logically related subsystems.

**After:** Modern DCS consoles offer 6-8 screens per operator, but operators must navigate hundreds of graphics pages. The "keyhole effect" occurs when operators focus on diagnosing one unit and miss developing problems in another. A documented case showed an operator navigating through 10 screens during a single alarm event while the process operated outside safe limits.

**Quantified cost:** Data overload and keyhole navigation cost companies millions per year. Operators become reactive -- waiting for alarms instead of detecting abnormal situations developing. If operators wait for alarms before acting, it extends the time operating outside limits. A shift with 100 alarms averaging 3 minutes each means 5 hours outside safe operating parameters.

**Solution direction:** The industry is now re-introducing "big picture" displays -- large video walls or Level 1 overview displays -- to restore what the analog panels provided inherently.

#### 2. Spatial Encoding and Muscle Memory

**Before:** Physical controls had fixed spatial positions. An operator's hand "knew" where the reactor coolant pump switch was -- they could reach for it without looking (Rasmussen's "skill-based" behavior level). The spatial layout of the panel was itself a cognitive map of the process.

**After:** Soft controls (NUREG/CR-6635) are software-mediated, context-dependent, and accessed through navigation. The same screen area may control different equipment depending on which page is displayed. This eliminates skill-based (automatic) interaction and forces operators into rule-based or knowledge-based modes, which are slower and more error-prone.

**New error modes introduced:**
- **Navigation errors**: Going to the wrong screen, losing track of current position in display hierarchy
- **Mode confusion**: A control action produces different results depending on the active mode
- **Context-dependent behavior**: The meaning of a button changes based on software state
- **Loss of peripheral awareness**: Cannot scan adjacent physical instruments while focused on one screen

#### 3. Rasmussen's SRK Framework Applied

Jens Rasmussen's Skills-Rules-Knowledge model (1983, developed after Three Mile Island) explains why digital transitions increase cognitive load:

- **Skill-based**: Automatic, pre-programmed motor patterns. Analog panels supported this via fixed spatial layout and tactile feedback. Digital screens largely eliminate this level.
- **Rule-based**: "If X, then do Y" -- following stored procedures. Digital systems can support this via on-screen procedures, but navigation overhead reduces performance.
- **Knowledge-based**: Required for unfamiliar situations, developing and testing hypotheses. Digital systems can help (access to more data) or hinder (information overload, inability to form a mental model of the complete system state).

The TMI operators failed because they were operating at the rule level when the situation demanded knowledge-based reasoning, and the analog panel layout did not help them understand the actual reactor state. However, the solution is not simply "go digital" -- it's designing digital interfaces that support all three cognitive levels.

#### 4. Endsley's Situation Awareness Model Applied

Mica Endsley's three-level SA model maps directly to the analog/digital transition:

- **Level 1 (Perception)**: "What are the current states?" Analog panels provided this inherently via spatial layout. Digital panels require navigation, creating gaps.
- **Level 2 (Comprehension)**: "What does it mean?" Pattern recognition across multiple indicators. Analog panels supported this via spatial proximity. Digital screens fragment related information across pages.
- **Level 3 (Projection)**: "What will happen next?" Trend recognition. Strip chart recorders provided inherent trend history. Digital trend displays exist but require explicit navigation to access.

#### 5. What Analog Patterns Persist and Should Be Preserved

**Analog objects in digital displays.** The high-performance HMI movement deliberately re-introduces analog metaphors:
- Bar graphs instead of numeric-only displays (faster perception at Endsley Level 1)
- Trend sparklines embedded inline (projection support at Level 3)
- Spatial grouping by process function (maintaining the "process topology as layout" principle)
- Color coding matched to physical equipment labeling (maintaining cross-reference between field and control room)

**Overview displays as "digital big boards."** ISA-101 Level 1 displays and large video walls are the digital successors to analog big boards, but they must be thoughtfully designed -- simply putting traditional graphics on a big screen does not restore situational awareness. The content must be specifically designed for distance viewing with minimal detail and maximum signal.

**Fixed-position critical displays.** Some information must never be navigated to -- it must always be visible. The nuclear SPDS (Safety Parameter Display System) is always displayed on the large panel. This principle persists in modern control rooms as dedicated "status bars" or "persistent panels."

## Cross-Cutting Synthesis: Design Principles for Meadowsyn

### Principle 1: Exception-Driven Attention (Cybersyn + Nuclear HMI)
The default display state should be calm and gray-scale. Color, motion, and sound are reserved exclusively for anomalies. An operator glancing at the screen should instantly know: is everything normal (gray, calm) or is something wrong (color, motion)? The Cybersyn algedonic model and the ASM Consortium's "boring display" principle converge on this: **design for the abnormal, not the normal.**

### Principle 2: Persistent Overview with Shallow Drill-Down (NASA + Nuclear + ATC)
NASA's Big Boards, the nuclear SPDS, and ATC's always-visible radar scope all share a principle: the macro-state view is always present. Meadowsyn should always show factory-level health. Detail views overlay or augment the overview; they never replace it. Maximum 2 clicks to any agent detail (ASM shallow navigation + ISA-101 hierarchy).

### Principle 3: Tiered Alarm Architecture (NASA + EEMUA + ATC)
Three tiers minimum: Advisory / Caution / Warning (NASA) or Low / Medium / High (EEMUA). Each tier has a defined color, defined operator response, and defined escalation timeout. Rate-limit to EEMUA targets: less than 1 alert per 10 min normal, less than 10 per 10 min during incidents. Auto-expire stale alerts. Support shelving with auto-restore.

### Principle 4: Spatial Stability and Predictability (Analog Persistence + Rasmussen SRK)
Agent positions should be spatially stable -- an operator who looks away and looks back should find the same agent in the same position. Layout should encode logical structure (team, project, pipeline phase). Interaction targets (buttons, shortcuts) should never move. This supports Rasmussen's skill-based behavior level, enabling automatic operation for routine tasks.

### Principle 5: Predictive Indicators Before Thresholds (ATC STCA + Cybersyn Algedonic)
ATC provides 50-90 seconds of warning before separation loss. Cybersyn's Cyberfilter detected incipient instability before it manifested as a crisis. Meadowsyn should show trajectory indicators (trend arrows, sparklines, burn-rate projections) that let operators see problems developing before hard thresholds are crossed. The ASM Consortium's 380% improvement in pre-alarm detection validates this approach.

### Principle 6: Dark-First, Color-Disciplined Palette (Control Room Standards + WCAG)
Off-black background (#121212), off-white text (#E0E0E0), tonal elevation hierarchy. Saturated colors are alarm indicators only. All normal-state rendering uses gray-scale. Meet WCAG 4.5:1 contrast for all text. This converges from three independent sources: nuclear HMI "going gray," control room ISO 11064 anti-glare standards, and web accessibility WCAG requirements.

### Principle 7: Bilateral Handoff Protocol (ATC Sector Handoff + NASA Go/No-Go)
When agents transfer work (context, PR review, dependent tasks), both the sending and receiving agent states should be visually indicated with a handoff animation. The receiving agent must "accept" the handoff. This prevents work from falling through cracks, directly adapted from ATC's flashing-data-block bilateral handoff protocol.

### Principle 8: Operative, Not Persuasive (Bonsiepe)
Every pixel must encode operational state. No decorative elements in the live operational view. No marketing language, no motivational metrics, no gamification. The interface exists to enable effective action, not to impress or persuade. Save aesthetic expression for the landing page; the operations view is pure function.
